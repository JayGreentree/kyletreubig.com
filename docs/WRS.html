1 Introduction

1.1 Purpose of This Document

The purpose of this document is to lay out the high-level requirements to which the design of kyletreubig.com must conform.

Top

1.2 Scope of This Document

This document describes the high-level customer requirements and derived requirements for kyletreubig.com. Requirements are broken down into functional, aesthetic, UI, and coding (or design) requirements.

Top

1.3 Description of This Document

The following section describes, at a high level, what kyletreubig.com is and how the user will interact through its pages. The proceeding section contains all requirements that must be met in the design of kyletreubig.com. Lastly, a requirements matrix is provided in the Appendix.

Top

2 General Description

2.1 Overview

The kyletreubig.com will be a resource to fellow web developers through informative articles and demonstration of design and development techniques.

Additionally, kyletreubig.com will be home to a personal portfolio of audio recordings.

Top

2.2 Goals

The goals of kyletreubig.com include, but are not limited to:

  • Provide an online presence for Kyle Treubig
  • Provide an outlet for both creative writing and the sharing of technical knowledge related to the design and development of websites
  • Provide an outlet for sharing musical audio recordings and receive feedback on recordings

Top

2.3 Target Audience

Kyletreubig.com is directed towards an audience that fully embraces modern usage of the internet. As such, there is no hindrance towards the use of the latest technologies, such as HTML 5 and JavaScript.

Tier 1 Audience
The tier 1 audience is friends and known acquaintances. While this audience may not be visiting kyletreubig.com for the technical content, they will visit out of acquaintance.
Tier 2 Audience
The tier 2 audience is classmates, coworkers, and fellow web developers. In short, anyone with a technical background and still an acquaintance.
Tier 3 Audience
The tier 3 audience is global web developers. This includes any visitors to the site referred by search engines or sister sites.

3.1.3 User Management Functions

  1. Privileged users can create new users
    1. The given password must be hashed with a salt
  2. Privileged users can modify user information
    1. If given, the new password must be hashed with a salt
  3. Privileged users can promote other users
  4. Privileged users can delete other users
  5. Users may log in with a username and password
  6. Users may log out
  7. Users may modify their own user information
  8. Users may change their password
    1. The given password must be hashed with a salt

Top

3.1.4 Content Management Functions

  1. Privileged users can edit page contents
  2. Modifications to page contents will be tracked
  3. Privileged users can compare versions of pages
  4. The about page will be editable and under version control
  5. Privileged users can upload image files
  6. Uploaded images can be included on editable pages

Top

3.1.5 Blog Functions

  1. Privileged users can create new articles
  2. Article content can be uploaded via a file or defined by direct input into a text editor
  3. Privileged users can change the state of articles
    1. The states are draft, published, archived
  4. Articles will belong to a single category
  5. Privileged users can create new categories
  6. Privileged users can rename categories
  7. Privileged users can delete categories
  8. Articles belonging to a deleted category become uncategorized
    1. The "uncategorized" category must exist and cannot be modified or deleted
  9. Articles can be tagged with multiple tag keywords
  10. Privileged users can edit draft and published articles
  11. A public RSS feed will display the last 10 published articles
  12. Published article view counts will be recorded
  13. View counts from the past seven days will determine popular articles
    1. Daily counts must be recorded for at least the past seven days
  14. Users can post comments on articles
  15. Guests will provide user information along with comment
  16. User profile information will be used for logged-in user's comments
  17. A daily email will contain a list of all new comments posted
  18. Privileged users can edit comments
  19. Privileged users can delete comments
  20. Privileged users can disable comments for individual articles
  21. Comments will be automatically disabled on articles over 90 days old
  22. Comments will be filtered by a Bayesian probability filter
    1. Comments with a Bayesian probability less than 0.2 are posted
    2. Comments with a Bayesian probability between 0.2 and 0.9 are placed in a moderation queue
    3. Comments with a Bayesian probability greater than 0.9 is classified are spam
  23. Privileged users can classify comments as clean or spam
    1. Privileged users can approve comments in the moderation queue
    2. Privileged users can classify comments in the moderation queue as spam
    3. Privileged users can classify posted comments as spam
    4. Privileged users can approve comments that have been classified as spam
  24. Privileged users can teach the Bayesian filter
    1. When approving comments, privileged users may choose to teach as ham
    2. When approving comments, privileged users may choose to unteach as spam
    3. When classifying as spam, privileged users may choose to teach as spam
    4. When classifying as spam, privileged users may choose to unteach as ham

Top

3.1.6 Music Portfolio Functions

  1. Privileged users can create new portfolio entries
  2. Privileged users can upload audio recordings for each portfolio entry
  3. Users can post comments on entries
  4. Portfolio comment requirements are the same as the blog comment requirements
  5. Privileged users can edit portfolio entries
  6. Privileged users can remove audio recordings from a portfolio entry
  7. Privileged users can delete portfolio entries
    1. All associated audio recordings must be deleted

3.5 Special Requirements

3.5.1 Access Control

In accordance with ACL convention, all access is denied unless explicitly specified as an access control rule

  1. Roles
    1. Guests are visitors who are not logged in
    2. Users are visitors who are logged in
    3. Admins are privileged users
  2. Rules
    1. Users can only modify user information for themselves
    2. Admins can perform all user management actions with exceptions:
      1. Admins cannot delete themselves
      2. Admins cannot change their own role (demote to lower role)
    3. Admins can perform all page management actions
    4. Admins can perform all assets management actions
    5. Admins can perform all blog management actions
    6. Admins can perform all music portfolio management actions
    7. All users can view published articles
    8. All users can post comments on articles
    9. Users cannot change their username

Top

3.5.2 Usability

  1. All front-end functionality must be usable when JavaScript is disabled
    1. All JavaScript functionality must degrade gracefully
  2. All front-end functionality must be usable in a text-only browser
    1. All content and navigation must be clearly displayed
